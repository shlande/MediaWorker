  narrow EventBroadcaster interface (Broadcast(eventType, payload) error),
  satisfied by *syncbroadcaster.SyncBroadcaster.
- AdminAccountsWriter extended with Ban/Unban (todo 6 registry semantics:
  Ban writes account_health + BAN event; Unban clears + UNBAN event).
- rotate: POST /v1/admin/accounts/{vendor}/{id}/rotate — request body IS the
  vendor auth field set; routed through todo 26's SAME ApplyAuthPatch path
  (read → OverlayAuthPatch → ValidateAuth → write changed → exactly ONE
  OnCredentialChange → CREDENTIAL_UPDATE broadcast carrying the new
  credential+client_config). No duplicated logic. Empty body/{} → 400
  field_errors.auth="required"; B4 failure → 400; missing account → 404.
- ban: body {reason?, ban_until?}; ban_until RFC3339, default now+24h
  (defaultBanDuration); empty body accepted (io.EOF tolerated); bad
  ban_until → 400 field_errors.ban_until. No second confirmation (UI's job).
- unban: registry.Unban(vendor, id).
- circuit: body {action:"force_open"|"force_close"}; other values → 400
  field_errors.action. Broadcasts types.EventCircuitForceOpen/Close with
  types.CircuitPayload{Vendor, AccountID} DIRECTLY via the injected
  broadcaster. Does NOT write account_health (node-local semantics) — locked
  by TestAccountOps_CircuitEdgeCases/no_account_health_write. Nil
  broadcaster → 500 "circuit broadcaster not configured", no panic.
- All four respond 202 {vendor, account_id, effective:"propagating"}
  (rotate also carries warnings when present). 202 = write/event accepted;
  nodes apply via todo 9's dispatcher.
- validateAccountPath helper extracted from todo 26's PUT handler and reused
  by all four ops handlers (vendor enum + account_id shape → 400).
- Zero secret leak preserved: ops responses echo only vendor/account_id/
  effective/warnings; rotate response grepped for sentinel values.
